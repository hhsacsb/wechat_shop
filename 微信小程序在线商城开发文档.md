# 微信小程序在线商城开发文档

## 1. 项目概述

### 1.1 项目名称
在线商城微信小程序

### 1.2 项目背景
为企业或商家搭建一套基于微信生态的在线商城系统，用户可通过微信小程序完成商品浏览、搜索、加购、下单、支付、订单查询、售后申请等操作；运营人员可通过管理后台完成商品、订单、用户、营销活动等管理。

### 1.3 项目目标
- 建立一套可稳定运营的微信小程序商城
- 实现商品展示、购物车、下单支付、订单管理等核心电商闭环
- 提供后台商品管理、订单管理、用户管理、营销管理能力
- 为后续扩展会员、积分、分销、秒杀、拼团等功能预留空间

### 1.4 一期建设原则
- 优先保证核心交易链路完整
- 优先保证页面加载速度和支付稳定性
- 优先保证后台运营配置能力
- 架构上兼顾后续功能扩展与维护成本

## 4. 业务流程

### 4.1 用户下单主流程
1. 用户进入小程序首页
2. 通过首页推荐、分类或搜索进入商品详情
3. 选择规格并加入购物车或立即购买
4. 在确认订单页选择地址、优惠券、配送方式
5. 提交订单并发起微信支付
6. 后端接收支付结果并更新订单状态
7. 商家在后台发货
8. 用户确认收货，订单完成

### 4.2 售后流程
1. 用户进入订单详情页
2. 提交退款或退货申请
3. 后台审核售后申请
4. 审核通过后执行退款或退货处理
5. 系统更新售后状态并通知用户

## 7. 技术架构设计

### 7.1 总体架构
系统采用以下架构模式：

`微信小程序前端 + 管理后台 Web + 后端 API 服务 + MySQL + Redis + 对象存储 + 微信生态能力`

### 7.2 架构分层
- 表现层：微信小程序、运营管理后台
- 接口层：RESTful API
- 业务层：用户、商品、订单、支付、营销、售后等服务
- 数据层：MySQL、Redis、对象存储
- 集成层：微信登录、微信支付、订阅消息、物流查询

### 7.3 推荐技术栈

#### 前端
- 微信原生小程序，适合快速上线
- 或 Taro，适合后续多端扩展
- UI 组件库建议使用 TDesign

#### 后端
- Node.js + NestJS + MySQL + Redis

#### 管理后台
- React + Ant Design

### 7.4 技术选型建议
- 如果团队偏 Java，优先选择 `Spring Boot`，成熟稳定，适合商城业务
- 如果团队偏前端全栈，优先选择 `NestJS`，开发效率较高
- 一期建议采用单体架构，避免过早拆分微服务增加复杂度

## 10. 小程序关键实现要点

### 10.1 登录方案
- 小程序通过 `wx.login` 获取 code
- 后端根据 code 换取 `openid` 和 `session_key`
- 首次登录自动创建用户信息
- 手机号获取使用微信授权能力

### 10.2 支付方案
- 用户提交订单后，后端统一生成待支付订单
- 后端调用微信支付接口生成预支付参数
- 小程序调起微信支付
- 后端接收支付回调并更新订单状态
- 支付回调必须进行签名校验和幂等处理

### 10.3 库存方案
- 下单时校验库存
- 支付成功后确认扣减库存，或下单时预扣库存
- 超时未支付订单自动关闭并回补库存
- 高并发场景下建议配合 Redis 锁或原子扣减方案

### 10.4 消息通知
- 下单成功通知
- 发货通知
- 售后进度通知
- 活动或优惠提醒

## 11. 性能与安全要求

### 11.1 性能要求
- 首页首屏加载时间控制在 2 至 3 秒内
- 商品列表采用分页加载
- 分类、配置项等静态数据使用缓存
- 热门商品、活动商品优先接入 Redis 缓存
- 商品详情图片采用压缩和懒加载
- 订单提交接口必须防重复提交

### 11.2 安全要求
- 所有业务接口必须鉴权
- 用户只能访问自己的订单、地址和售后数据
- 支付回调必须验签
- 后台接口必须做角色权限控制
- 敏感数据如手机号应脱敏展示
- 关键操作必须记录日志

## 12. 异常处理方案

### 12.1 前端异常处理
- 统一处理网络异常提示
- 统一处理接口失败提示
- 页面提供加载中、空状态、失败重试等交互
- 支付失败后支持重新支付

### 12.2 后端异常处理
- 统一错误码规范
- 使用全局异常拦截
- 订单、支付、库存等关键流程记录业务日志
- 第三方接口超时支持重试和降级

## 16. 项目风险与应对

### 16.1 需求变更风险
- 风险：需求频繁变化影响开发进度
- 应对：冻结一期范围，二期功能延后实施

### 16.2 交易链路风险
- 风险：订单、库存、支付之间存在一致性问题
- 应对：设计清晰的订单状态机与库存扣减机制

### 16.3 平台审核风险
- 风险：微信审核不通过影响上线时间
- 应对：提前规避敏感类目、违规内容和高风险描述

### 16.4 高并发风险
- 风险：活动期间流量突增导致系统压力过大
- 应对：增加缓存、限流、压测和热点数据隔离
